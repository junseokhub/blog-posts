# Outbox vs CDC vs Saga

분산 시스템에서 DB에 쓰면서 그 변경을 메시지로도 내보내야 하는 상황은 끝없이 나온다. 이걸 푸는 길로 **Outbox**, **CDC**, **Saga**가 자주 같이 거론된다. 
먼저, 셋은 같은 층의 경쟁자가 아니다. 둘은 진짜 양자택일이고, 하나는 아예 다른 층의 문제를 푼다.

이 글은 셋이 각각 무슨 문제를 푸는지, 어디가 같은 선택지고 어디가 다른 층인지 가른다.

---

## 셋은 같은 층이 아니다

여기 섞여 있는 건 사실 두 개의 다른 질문이다.

1. 한 서비스에서 DB 변경을 어떻게 유실 없이 이벤트로 내보내나 —> 발행 보장
2. 여러 서비스에 걸친 하나의 작업을 어떻게 일관되게 끝내나 —> 분산 트랜잭션

Outbox와 CDC는 1번을 푸는 두 가지 방법이다. Saga는 2번을 푼다. 그리고 결정적으로, Saga는 1번(발행 보장)을 깔고 돈다. 그래서 층이 이렇게 쌓인다.

```
[흐름 일관성 층]   Saga          <- 여러 서비스 조율
                    │ 단계를 잇는 이벤트가 유실되면 멈춘다
                    │ 그래서 신뢰성 있는 발행이 깔려야 함
[발행 보장 층]     Outbox  또는  CDC
```

이 그림만 머리에 박으면 나머지는 따라온다.

---

## 발행 보장이라는 문제

본론에 들어가기 전에 1번 문제부터 짧게 못 박자. 
한 작업에서 DB에 쓰고, 그 사실을 브로커(Kafka 등)로도 발행해야 한다고 하자.

```java
@Transactional
public void doSomething(...) {
    repository.save(entity);   // 1. DB
    kafkaTemplate.send(...);   // 2. Kafka — DB 트랜잭션 밖
}
```

DB 트랜잭션은 DB까지만 책임진다. Kafka는 그 바깥이라 트랜잭션에 안 들어온다. 그래서 1번 커밋 후 2번 직전에 죽으면 이벤트는 영영 안 나가고, 2번을 먼저 보냈는데 트랜잭션이 롤백되면 없는 데이터의 후속 처리가 일어난다. 두 저장소에 따로 쓰는 한 이 어긋남은 구조적으로 생긴다.

발행 보장은 이 어긋남을 없애는 문제다. Outbox와 CDC가 이걸 서로 다른 방식으로 푼다.

---

## Outbox vs CDC

둘 다 DB 변경을 유실 없이 이벤트로 내보낸다. 차이는 이벤트를 어디서 만드는지이다.

### Outbox (애플리케이션이 직접 적는다)

이벤트를 브로커로 바로 보내지 않고, 같은 DB에 같은 트랜잭션으로 outbox(보낼 편지함) 테이블에 먼저 적는다. 비즈니스 데이터와 이벤트가 한 트랜잭션에 들어가니 같이 커밋되거나 같이 롤백된다. 발행은 별도 폴러가 그 테이블을 읽어서 처리한다. 이벤트는 애플리케이션이 의도적으로 만든다.

```java
@Transactional
public void doSomething(...) {
    repository.save(entity);
    outboxRepository.save(Outbox.of("event.type", payload));  // 같은 트랜잭션
}
```

### CDC (DB의 로그를 읽는다)

CDC(Change Data Capture, 변경 데이터 캡처)는 DB가 모든 변경을 순서대로 남기는 트랜잭션 로그(MySQL의 binlog, PostgreSQL의 WAL, MongoDB의 Oplog)를 별도 도구(Debezium 등)가 따라 읽어서, 변경을 이벤트로 바꿔 브로커로 흘려보낸다. 애플리케이션은 평소처럼 DB에 쓰기만한다. 이벤트화는 DB 바깥에서 일어난다.

> 참고로 MongoDB의 Change Streams라는 자체 내장 기능이 있는데, Oplog기반이지만, Debezium없이도 가능하다.

```
[Outbox]  앱: 비즈니스 + outbox 적재(한 트랜잭션) -> 폴러 -> 브로커
[CDC]     앱: 비즈니스 쓰기만 -> DB 트랜잭션 로그 -> 커넥터 -> 브로커
```

### 갈리는 지점

- 코드 침투
  - Outbox는 outbox 테이블·적재 코드·폴러·복구를 직접 만든다. CDC는 앱 코드가 거의 안 바뀐다. 그냥 쓰던 대로 DB에 쓴다
- 추가 인프라 
  - Outbox는 0이다. 트랜잭션 되는 DB만 있으면 된다. CDC는 커넥터(Debezium)와 그걸 돌릴 런타임(보통 Kafka Connect)이라는 운영해야 할 컴포넌트가 붙는다
- 이벤트의 모양
  - Outbox는 도메인 사건을 의도적으로 설계해 적는다. CDC가 기본으로 주는 건 행 변경(어떤 테이블의 이 row가 이렇게 바뀜)이다. 도메인 의미로 쓰려면 가공이 필요하다
- 스키마 결합 
  - CDC는 테이블 스키마에 묶인다.컬럼을 바꾸면 흘러나가는 이벤트가 따라 바뀐다. Outbox는 payload를 앱이 통제하므로 내부 스키마와 분리된다
- 지연 
  - Outbox 폴링은 폴링 주기만큼 지연이 깔린다. CDC는 로그를 따라가므로 보통 더 낮은 지연을 낸다

판단은 단순하다.
추가 인프라 없이 시작하고 이벤트를 직접 설계하고 싶으면 Outbox. 앱 코드를 못 건드리는 상황(레거시 DB에서 이벤트를 뽑아야 한다든가)이거나 이미 CDC 인프라가 있으면 CDC를 선택하면 된다.

> 하이브리드 - outbox 테이블에 의도적으로 적고, 그 테이블을 CDC로 읽는 방식도 있다. 도메인 이벤트 설계(Outbox의 장점)와 폴러 제거·저지연(CDC의 장점)을 같이 가져간다. 인프라 여력이 되면 깔끔한 선택지다.

두 방법 모두 at-least-once다.
유실은 막지만 같은 이벤트가 두 번 나갈 수 있다. 그래서 받는 쪽 멱등성은 어느 쪽을 골라도 똑같이 필요하다.

---

## Saga

Outbox/CDC는 한 서비스에서 이벤트를 내보내는 이야기였다. Saga는 그 위층이다.여러 서비스에 걸친 하나의 작업을 어떻게 일관되게 끝내느냐를 푼다.

문제의 뿌리는 이거다. 분산 환경엔 여러 서비스를 한 번에 묶는 ACID 트랜잭션이 없다.서비스 A가 커밋하고, B가 커밋하고, C가 실패해도, A와 B를 한 번에 되돌릴 전역 롤백이 없다. 각 서비스의 DB는 자기 트랜잭션까지만 책임진다.

Saga의 답은 이렇다. 큰 작업을 각 서비스의 로컬 트랜잭션 여러 개로 쪼개고, 각 단계가 다음 단계를 이벤트로 트리거한다. 중간에 실패하면 전역 롤백 대신, 이미 끝난 단계를 거꾸로 상쇄하는 보상 트랜잭션을 돌려 되돌린다.

```
정상:  A 커밋 -(이벤트)->  B 커밋 -(이벤트)-> C 커밋
실패:  A 커밋 ---------> B 커밋 ---------> C 실패
보상:  A 보상 <--------- B 보상 <------------
```

단계를 잇는 그 이벤트가 유실되면 Saga는 중간에 멈춘다.A는 커밋됐는데 B를 깨우는 이벤트가 안 나가면, 작업은 절반만 된 채 영영 멈춘다. 그래서 Saga의 각 단계는 이벤트를 신뢰성 있게 내보내야하고, 그게 Outbox/CDC가 하던 일이다. Saga가 Outbox/CDC를 쓴다.

> Saga를 어떻게 조율하느냐(중앙 지휘 vs 각자 반응), 보상을 어떻게 설계하느냐는 더 깊게 다루는 글이 따로 있다.

---

## 그래서??

- Outbox vs CDC
  - 같은 발행 보장 문제의 두 방법이고, 보통 하나를 고른다
- Outbox/CDC vs Saga 
  - 층이 다르다. Saga는 발행 보장을 전제로 도는 상위 패턴이다. Saga를 쓴다라는 것은 보통 그 밑에 Outbox나 CDC를 까는 것을 포함한다.

---

## 언제 뭘 써야지?

무조건적으로 정해진건 아니지만 일반적으로 정리하면 아래와 같다.

- Outbox
  - 한 서비스의 DB 변경을 이벤트로 처리하거나, 추가 인프라 없을 때
- CDC
  - 앱 코드를 안 건드리고 DB(레거시 포함)에서 이벤트를 뽑아야하거나 이미 CDC 인프라가 있을 때
- Saga (밑에 Outbox/CDC를 깔고)
  - 여러 서비스에 걸친 작업의 일관성이 필요하고 전역 롤백이 없을 때

---

## 장단점

**Outbox**
- 장점 
  - 추가 인프라 0, 이벤트 payload를 앱이 통제.
- 단점
  - 코드 침투(테이블·폴러·복구), 폴링 지연.

**CDC**
- 장점
  - 앱 코드 무침투, 보통 더 낮은 지연.
- 단점
  - 커넥터 인프라 운영 부담, 테이블 스키마에 결합, 행 변경을 도메인 이벤트로 가공해야 함.

**Saga**
- 장점
  - 전역 트랜잭션 없이 여러 서비스의 일관성을 확보.
- 단점
  - 복잡도가 크게 오름, 보상 트랜잭션 설계 부담, 최종 일관성이라 중간 상태가 잠깐 노출된다.

---

## 정리

- 이벤트를 어떻게 내보내나?(발행 보장) vs 여러 서비스를 어떻게 일관되게(분산 트랜잭션)
- Outbox·CDC 
  - 발행 보장의 두 방법. 
  - Outbox는 앱이 적고, CDC는 DB 로그를 읽는다. 진짜 양자택일이다.
- Saga
  - 상위 층. 여러 서비스의 흐름을 로컬 트랜잭션 + 보상으로 잇고, 그 밑에 Outbox/CDC를 깐다.
- 셋 다 결국 at-least-once 위에서 돌므로, 받는 쪽 멱등성은 공통 숙제다.

```
발행 보장 ---> Outbox (앱이 적는다)
             -> CDC    (DB 로그를 읽는다)
                 -> Saga 가 이 위에서 여러 서비스를 조율한다
                       -> 단계 되돌리기는 "보상 트랜잭션"
```

## 마무리
Outbox를 써야하나 CDC를 써야하냐는 한 서비스 안에서 데이터를 어떻게 흘려보낼지 고르는 기술적 방법론이고, Saga를 쓸까?는 여러 서비스 간의 비즈니스 트랜잭션을 어떻게 관리할지 설계하는 아키텍처 패턴이다. 이 선을 정확히 그으면 셋이 더는 헷갈리지 않는다.

결국 중요한 것은 내 시스템의 상황에 맞게 언제, 어떻게 쓸지 올바르게 판단하는 것이다.
추가 인프라 비용 없이 비즈니스 의미가 담긴 이벤트를 안전하게 발행하고 싶다면 Outbox를,

애플리케이션 코드 손상 없이 실시간으로 대량의 데이터 변경을 추적하고 싶다면 CDC를 선택하면 된다.

그리고 이렇게 만들어진 이벤트들을 엮어 거대한 분산 시스템의 정합성을 맞춰야 하는 순간이 오면, 그때 비로소 Saga 패턴이라는 큰 그림을 그리면 된다.

트레이드오프를 명확히 이해하고 현재 우리 팀의 인프라 성숙도와 비즈니스 요구사항에 맞는 최적의 조합을 찾아야 한다.
